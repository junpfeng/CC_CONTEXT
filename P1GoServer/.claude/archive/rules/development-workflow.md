# 游戏功能业务开发流程规范

使用 Claude Code 进行游戏功能开发的标准流程。

---

## 流程概览

```
需求分析 → 方案设计 → 代码实现 → 构建验证 → 测试验证 → 代码提交
```

---

## 阶段 1: 需求分析

**目标**: 理解需求，明确边界

**输入要求**:
- 描述清楚业务场景和预期行为
- 说明涉及的实体（玩家、NPC、物品等）
- 明确输入输出和触发条件
- 说明与现有系统的关联

**示例输入**:
```
我需要给 NPC 添加一个"收集资源"的行为：
- NPC 自动寻找附近 10 米内的资源点
- 走过去采集（耗时 3 秒）
- 带回最近的仓库存放
- 仓库满时停止采集
```

**Claude 行为**:
- 理解需求的业务含义
- 识别可能的技术难点
- 提出澄清问题

---

## 阶段 2: 方案设计 (Plan Mode)

**目标**: 探索代码库，设计实现方案

**触发方式**:
```
帮我设计 XXX 功能的实现方案
```

**Claude 自动执行**:

| 步骤 | 行为 |
|------|------|
| 探索现有代码 | 搜索相关模块、理解现有架构 |
| 找参考实现 | 寻找类似功能的实现模式 |
| 设计方案 | 输出实现计划，包括涉及的文件和改动点 |
| 等待确认 | 用户审核后才开始实现 |

**方案文档输出内容**:
- 涉及的文件列表
- 每个文件的改动说明
- 新增文件说明（如需要）
- 潜在风险和注意事项
- 依赖的前置条件

**用户决策点**:
- 确认方案 → 进入实现阶段
- 调整方案 → 继续讨论
- 拒绝方案 → 重新设计

---

## 阶段 3: 代码实现

**目标**: 按计划逐步实现

**推荐方式**:

```bash
# 方式1: 批量实现（适合小功能）
按照计划开始实现

# 方式2: 分步实现（适合大功能，更可控）
先实现第一步：定义数据结构
继续实现第二步：编写核心逻辑
继续实现第三步：添加接口调用
```

**Claude 遵循原则**:
- 先读取相关文件，理解上下文
- 遵循项目代码风格（见 `go-style.md`）
- 不修改自动生成的文件
- 保持改动最小化，不过度工程化
- 不添加不必要的注释、文档、错误处理

**禁止行为**:
- 未读取文件就修改
- 添加超出需求范围的功能
- 修改 `cfg_*.go`、`*_pb.go` 等自动生成文件
- 添加过度的错误处理和边界检查

---

## 阶段 4: 构建验证

**目标**: 确保代码可编译

**执行方式**:
```bash
# 使用构建技能
/build-server

# 或指定服务器
帮我构建 scene_server 检查是否有编译错误
```

**Claude 行为**:
- 执行 `make <server>` 构建命令
- 分析编译错误
- 自动修复错误
- 迭代直到构建成功

**常用构建命令**:
```bash
make build              # 构建所有服务器
make scene_server       # 构建场景服务器
make logic_server       # 构建逻辑服务器
make test               # 运行测试
```

---

## 阶段 5: 测试验证

**目标**: 验证功能正确性

**执行方式**:
```bash
# 运行相关测试
运行 bt 相关的单元测试

# 运行指定包测试
go test -v ./servers/scene_server/internal/common/ai/bt/...

# 运行全部测试
make test
```

**需要添加测试时**:
```
帮我为这个功能添加单元测试
```

**测试文件命名**: `xxx_test.go`

---

## 阶段 6: 代码提交

**目标**: 规范提交代码

**执行方式**:
```bash
/commit
```

**Claude 行为**:
1. 执行 `git status` 和 `git diff`
2. 生成符合项目规范的 commit message
3. 等待用户确认后提交

**Commit Message 格式**:
```
feat:[模块] 功能描述
fix:[模块] 修复描述
refactor:[模块] 重构描述
```

## 常见开发场景

### 场景 1: 添加 NPC 行为

```
1. /ai-decision flow          # 了解决策系统架构
2. 设计行为方案               # Plan Mode
3. /ai-decision add-plan xxx  # 添加计划
4. /ai-decision add-feature   # 添加特征值
5. /build-server              # 验证构建
6. /commit                    # 提交代码
```

### 场景 2: 添加行为树节点

```
1. 阅读 behavior-tree.md 规范
2. 设计节点方案               # Plan Mode
3. 实现节点代码
4. 注册到 factory.go
5. 添加 JSON 配置或代码定义
6. /build-server              # 验证构建
7. 运行测试
8. /commit                    # 提交代码
```

### 场景 3: 修复 Bug

```
1. 描述 Bug 现象
2. Claude 分析定位问题
3. 确认修复方案
4. 实现修复
5. /build-server              # 验证构建
6. 验证修复效果
7. /commit                    # 提交代码
```

## 最佳实践

### 沟通技巧

| 做法 | 效果 |
|------|------|
| 描述业务场景 | Claude 理解"为什么"，给出更合适方案 |
| 提供示例 | 减少歧义，加快理解 |
| 分步确认 | 大功能分多次实现，降低风险 |
| 及时反馈 | 发现问题立即纠正，避免返工 |

### 效率技巧

| 技巧 | 说明 |
|------|------|
| 善用 Plan Mode | 改动涉及 3+ 文件时先规划 |
| 保持小步迭代 | 复杂功能分多次对话完成 |
| 及时构建验证 | 每完成一个逻辑单元就构建一次 |
| 利用项目技能 | `/ai-decision`、`/build-server` 等 |

### 质量保障

| 检查项 | 说明 |
|--------|------|
| 代码风格 | 遵循 `go-style.md` |
| 构建通过 | `make build` 无错误 |
| 测试通过 | `make test` 无失败 |
| 提交规范 | commit message 格式正确 |
