# 动物系统技术设计审查

> **审查日期**: 2026-03-23 | **结论**: 有条件通过（2 个严重问题需修改，4 个建议改进）

## 严重问题（必须修改）

### S1: BtTickSystem 管线路由架构不匹配

**位置**: technical_design.md 4.1 / server.md 2.2

**问题**: 设计称"BtTickSystem 新增 `animalPipeline *OrthogonalPipeline` 字段，根据 ExtType 分发"。但当前 `BtTickSystem`（`bt_tick_system.go:27-33`）持有单一 `orthogonalPipeline`，在 `Update()` 中遍历所有 NPC 实体统一走这条管线。大世界场景中动物和人形 NPC 共存，需要两条管线并行。

当前代码 `Update()` 第 76-107 行的循环不做 ExtType 判断，所有 NPC 实体都走同一管线。如果简单加 `animalPipeline` 字段，需要在循环内按 `SceneNpcComp.ExtType()` 分叉路由。但这意味着：
1. `syncNpcMovement` / `syncNpcStateToAnimComp` / `syncNpcStateToTownNpc` 等后续同步也需要按 ExtType 分叉
2. 情绪系统 `emotion.Default.Tick` 对动物无意义，需跳过

**建议**: 不要在 BtTickSystem 上加第二个管线字段。改用现有 `v2PipelineRegistry` 的 map 模式——`SetupV2Pipeline` 已支持按 SceneType 注册。但 SceneType 是场景级别的，不是实体级别的。建议改为让 BtTickSystem 持有 `map[SceneNpcExtType]*OrthogonalPipeline`，在 `Update()` 循环中按实体 ExtType 查表分发。同时明确 Animal 实体跳过情绪 Tick 和 TownNpc 同步。

### S2: 客户端模块路径矛盾

**位置**: client.md 2.1 vs technical_design.md 5.1

**问题**: client.md 写的路径是 `Assets/Scripts/Gameplay/Modules/S1Town/Entity/Animal/`，而 technical_design.md 5.1 自己指出"动物属于大世界系统，应放 BigWorld"并写了 `BigWorld/Entity/Animal/`。两份文档互相矛盾。动物是大世界实体，不应放 S1Town。

**建议**: 统一为 `BigWorld/Entity/Animal/`，同步修改 client.md 2.1。

## 建议改进（推荐修改）

### R1: AnimalData.follow_target_id 下发给所有客户端存在隐私泄露

**位置**: protocol.md 3.1

**问题**: `follow_target_id`（玩家实体 ID）作为 AnimalData 的一部分随 NpcDataUpdate 广播给 AOI 内所有玩家。虽然当前无安全性需求，但从设计规范看，内部状态不必要地暴露给全部客户端。server.md 3.2 明确说 `AnimalPerceptionState` 是"内部状态，不下发客户端"，但 FieldAccessor 注册列表中又包含了 `Perception.FollowTargetID`，自相矛盾。

**建议**: 确定 `follow_target_id` 是否需要下发。如果客户端 FollowState 只需要知道"正在跟随"（由 AnimalState=Follow 已表达），则从 AnimalData 中移除 `follow_target_id`，FieldAccessor 也不注册。如果客户端需要知道跟随谁（用于朝向插值等），则保留但在 server.md 中修正描述。

### R2: 错误码跳段过大

**位置**: protocol.md 6.3

**问题**: 当前 Codes 最大值 10005，动物错误码直接跳到 14001-14004，中间空了约 4000 个号段。虽不影响功能，但暗示可能遗忘了中间是否有其他系统占用。

**建议**: 确认 10006-14000 段确实无占用。如果是有意预留，在文档中注明段位分配规则。

### R3: Chicken 感知系统浪费

**位置**: server.md 4.4 / 8.4

**问题**: Chicken 是纯静态装饰（moveSpeed=0，永远 Idle），但仍注册了 DistanceSensor（视觉 10m + 听觉 10m）。感知结果无任何消费者——没有逃跑、没有交互、没有状态切换。

**建议**: Chicken 不注册感知插件，或者在 AnimalSpawner 中对 Chicken 不创建感知组件，减少无谓计算。

### R4: 物品校验缺乏具体定义

**位置**: server.md 5.1 / protocol.md 5.1

**问题**: `AnimalFeedReq.item_id` 是 string 类型，校验描述为"物品为有效食物"，但未定义：哪些 item_id 是有效食物？是配置表驱动还是硬编码？背包扣减调用哪个接口？

**建议**: 补充食物物品的判定规则（配置表字段或物品标签），以及背包扣减的接口调用说明。

## 确认无问题的部分

1. **协议设计**: NpcDataUpdate 字段号 38 经确认最大为 37（tradeProxyInfo），无冲突。AnimalState 独立枚举不混用 NpcState，设计合理。全量快照 + 事件通知互补的同步策略清晰。
2. **SceneNpcExtType=4**: 当前 iota 到 TownGta=3，值 4 正确。
3. **事务设计**: 喂食流程在 scene_server 单线程内完成，无跨服事务，幂等性通过 FollowTargetID!=0 保证，设计正确。
4. **V2 管线复用**: `RegisterV2Pipeline` + `animalDimensionConfigs()` 的注册模式与现有 Town/Sakura/TownGta 一致，ExtHandler 模式可复用。
5. **客户端组件架构**: Controller+Comp 模式与 MonsterController 一致，精简到 6-7 个组件合理。EntityType.Animal 独立枚举避免了在 Npc 创建路径中的大量分叉。
6. **验收测试**: 7 个测试用例覆盖了生成、动画、飞行、喂食交互、同屏限制、LOD、异常场景，MCP 脚本验证方式可执行。
7. **需求覆盖**: 4 种动物的行为、交互、性能预算在设计中均有对应方案，无遗漏需求点。
8. **FieldAccessor/Snapshot 同步**: 设计明确提到需同步扩展（遵循 feedback_npcstate_snapshot_sync），列出了具体字段清单。
