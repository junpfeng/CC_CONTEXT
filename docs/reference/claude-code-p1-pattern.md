# Claude Code P1 模式：知识索引 + Agent 委托架构

## 一、设计哲学

**核心假设**：AI 是有判断力的协作者，不是需要硬约束的执行者。

三条原则：
1. **按需加载**：不把所有知识塞进 system prompt，用索引按需拉取
2. **委托而非命令**：把任务委托给专家 Agent，保留 AI 推理能力
3. **经验闭环**：项目记忆可积累、可追溯、可迭代

对比两种极端：
- 零配置：AI 每次从零开始，反复犯同样的错
- 全量灌入：所有规则/文档塞进 prompt，上下文窗口浪费，信噪比低

P1 模式的定位是**中间路线**：轻量骨架 + 按需加载 + 经验积累。

## 二、目录结构

```
.claude/
├── CLAUDE.md                  # 入口：项目概况 + 索引引用
├── index.md                   # 中枢：关键词 → 文档映射
│
├── rules/                     # 柔性规则（指导性，非强制）
│   └── *.md
│
├── skills/                    # 工作流技能
│   └── <workflow-name>/
│       └── *.md
│
├── agents/                    # 专家 Agent 定义
│   ├── go-expert.md
│   ├── debugger.md
│   ├── code-reviewer.md
│   └── go-feature-dev.md
│
├── docs/                      # 知识文档库
│   └── *.md
│
├── learned/                   # 经验积累（self-learning 产出）
│   └── *.md
│
└── over-plan/                 # 历史计划归档（项目记忆）
    └── *.md
```

## 三、各组件详解

### 3.1 CLAUDE.md — 入口

项目的第一入口，Claude Code 自动加载。内容应精简，职责：
- 项目一句话定位
- 技术栈概要
- 引用 `index.md` 作为知识查找入口
- 基本工作约定（如测试要求、提交规范）

**关键**：不在这里堆砌知识，只做导航。

### 3.2 index.md — 知识索引中枢

这是 P1 模式区别于其他方案的核心组件。

```markdown
# 知识索引

## 行为树
- 架构设计 → docs/bt-architecture.md
- 节点开发 → docs/bt-node-guide.md
- 经验教训 → learned/bt-lessons.md

## ECS 架构
- 组件规范 → docs/ecs-components.md
- System 编写 → docs/ecs-system-guide.md

## NPC AI
- 决策系统 → docs/decision-system.md
- Brain 配置 → docs/brain-config.md
```

工作方式：
1. AI 接到任务，先查 `index.md` 找相关关键词
2. 按需读取对应文档，只加载与当前任务相关的知识
3. 上下文窗口保持干净，信噪比高

### 3.3 rules/ — 柔性规则

与 P4 的"宪法（constitution）"不同，P1 的 rules 是**指导性**的：
- 遇到特殊情况可以灵活处理
- 不是"违反即失败"的硬约束
- 更像团队编码规范，而非法律条文

适合放入 rules 的内容：
- 代码风格约定（命名、日志格式）
- 架构边界（哪些包不能互相依赖）
- 常见陷阱提醒（如"Brain plan name 必须与 JSON 树 name 一致"）

### 3.4 agents/ — 专家委托

4 个 Agent 各有分工，形成协作链：

| Agent | 职责 | 触发时机 |
|-------|------|----------|
| go-expert | Go 代码实现 | 编码任务 |
| go-feature-dev | 功能开发（含子任务调度） | 新功能需求 |
| code-reviewer | 代码审查 | 实现完成后按需触发 |
| debugger | 问题定位与修复 | Bug 报告 |

设计要点：
- Agent 之间是**委托关系**，不是固定流水线
- 每个 Agent 有独立的 system prompt，定义其专长和行为边界
- 工作流中由主 Agent 判断何时调用哪个专家

与 P4 command 的对比：
- P4 command 是固定步骤脚本："第一步做 X，第二步做 Y"
- P1 agent 是角色委托："你是 Go 专家，请解决这个问题"
- P1 保留了 AI 的推理和判断能力，P4 更强调流程可控

### 3.5 skills/ — 工作流技能

Skills 是可复用的工作流模板，按功能域组织：

```
skills/
└── dev-workflow/
    ├── BTree.md          # 行为树开发流程
    └── ...
```

与 Agent 的区别：
- Agent 定义"谁来做"（角色）
- Skill 定义"怎么做"（流程）
- Agent 执行任务时可以参考 Skill 中的流程指导

### 3.6 docs/ — 知识文档库

存放项目的架构设计、接口文档、模块说明等。特点：
- 不自动加载，通过 `index.md` 按需引用
- 内容相对稳定，不频繁更新
- 面向 AI 编写：重点讲清约束和边界，而非面面俱到

### 3.7 learned/ — 经验积累

self-learning 机制的产出物。工作流程：

```
AI 发现可复用模式 → 询问用户确认 → 写入 learned/*.md → 更新 index.md
```

适合记录的内容：
- 反复出现的 Bug 模式及修复方式
- 项目特有的隐式约定（代码中看不出来的）
- 跨模块交互的注意事项

与 docs 的区别：
- docs 是**设计时知识**（架构、规范）
- learned 是**运行时经验**（踩过的坑、发现的规律）

### 3.8 over-plan/ — 历史计划归档

每次任务的计划归档，提供**项目记忆**能力：
- 新对话中可以回溯"之前为什么这样做"
- 追踪决策演变过程
- 避免重复讨论已经做过的决定

## 四、信息流转模型

```
用户任务
  │
  ▼
CLAUDE.md（入口，基本约定）
  │
  ▼
index.md（查索引，定位相关知识）
  │
  ├──→ docs/*.md（按需加载设计文档）
  ├──→ learned/*.md（按需加载经验教训）
  └──→ rules/*.md（加载相关规则）
  │
  ▼
选择 Agent（判断委托给谁）
  │
  ├──→ go-expert（编码）
  ├──→ code-reviewer（审查）
  ├──→ debugger（调试）
  └──→ go-feature-dev（功能开发）
  │
  ▼
Agent 参考 skills/（执行工作流）
  │
  ▼
任务完成
  │
  ├──→ over-plan/（归档计划）
  └──→ learned/（沉淀经验，需用户确认）
```

## 五、规模参考

P1 实际运行的组件规模：

| 组件 | 数量 | 增长频率 |
|------|------|----------|
| rules | ~7 | 低（稳定后很少新增） |
| agents | 4 | 极低（角色基本固定） |
| skills | ~7 | 中（新工作流时新增） |
| docs | ~18 | 中（新模块时新增） |
| learned | ~7 | 中（经验逐步积累） |
| over-plan | 60+ | 高（每次任务产出） |

总知识量不大，但通过索引机制实现了**高效访问**。

## 六、适用场景与局限

### 适合
- 长期迭代的后端项目（知识持续积累有价值）
- AI 参与度高的开发流程（Agent 委托能充分发挥）
- 模块化程度高的代码库（知识可以按域切分）

### 不适合
- 短期一次性项目（知识积累 ROI 低）
- 强合规要求的项目（柔性规则不够，需要 P4 式硬约束）
- 多人共享 AI 配置的场景（learned 带有个人偏好）

### 可以借鉴 P4 的改进方向
1. **经验自动固化**：bug-fix 流程中自动判断是否需要创建 skill/rule，减少对用户主动参与的依赖
2. **负面清单**：对高风险操作增加少量硬约束 rule（如"禁止 force push main"）
3. **知识可度量**：为 skill 增加评测机制（evals），验证知识积累的有效性
