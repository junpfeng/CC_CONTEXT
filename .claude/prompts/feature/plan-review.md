## 参数解析

用户传入的完整参数：`$ARGUMENTS`

**解析规则（按优先级尝试）：**

1. **自动匹配目录**：用 Glob 工具搜索 `docs/Version/*/` 下的子目录，将参数中的空格分隔词与目录结构匹配。典型用法是 `v0.0.2-mvp deep-research`，对应 `docs/Version/v0.0.2-mvp/deep-research/`
2. **斜杠分隔**：如果参数包含 `/`，按 `/` 拆分为 version_id 和 feature_name
3. **验证路径**：确认 `docs/Version/{version_id}/{feature_name}/plan.json`（或兼容旧版 `plan.md`）存在
4. **无法自动匹配时**：使用 AskUserQuestion 向用户确认，列出已有的版本目录和功能目录供选择

获得完整参数后再继续。

## 任务

对 `docs/Version/{version_id}/{feature_name}/plan.json`（及其子文件 `plan/*.json`；兼容旧版 `plan.md` + `plan/*.md`）进行深度 Review，找出遗漏、隐患和改进空间，确保 plan 足够完善，可以一把生成考虑周全的代码。

## 你的角色

1. **资深 Code Reviewer**：从实现可行性角度审查
2. **QA 工程师**：从测试和边界场景角度审查
3. **安全工程师**：从安全和防作弊角度审查
4. **运维工程师**：从部署、监控、可观测性角度审查

## 工作流程

### 第一步：加载上下文

**使用并行 Agent 加速上下文建立：**

- **Agent 1（读取 Plan）**：完整读取 `plan.json`（或旧版 `plan.md`）和 `plan/` 目录下所有子文件（.json 或 .md）。如果是简单方案（只有 plan.json），只读取 plan.json
- **Agent 2（读取需求）**：读取 `feature.json`（或旧版 `feature.md`）需求文档，理解原始需求
- **Agent 3（读取宪法）**：读取 `P1GoServer/.claude/rules/constitution.md` 和 `freelifeclient/.claude/constitution.md`，以及两端的 CLAUDE.md

等所有 Agent 返回后，综合建立完整上下文。

### 第二步：多维度 Review

从以下 **7 个维度** 逐一审查，每个维度给出具体问题和改进建议：

---

#### 维度 1：需求覆盖度

对比 `feature.json`（或 `feature.md`）和 `plan.json`（或 `plan.md`），检查：

- [ ] 需求文档中的每个功能点是否都在 plan 中有对应设计？
- [ ] 是否有需求被遗漏或只是一笔带过？
- [ ] 隐含需求是否被识别？（如：需求说"玩家可以交易物品"，plan 是否考虑了交易取消、交易超时、物品锁定？）

---

#### 维度 2：边界条件与异常场景

系统性检查以下边界情况：

**数据边界：**
- [ ] 空数据：列表为空、配置不存在、玩家数据未初始化
- [ ] 上限溢出：数量超过上限、背包满、等级到顶
- [ ] 并发修改：两个请求同时修改同一资源
- [ ] 非法输入：负数、零值、超大数字、非法枚举值

**时序边界：**
- [ ] 断线重连：操作进行到一半断线，重连后状态是否正确？
- [ ] 重复请求：客户端因网络抖动发送两次相同请求
- [ ] 操作时序：A 操作依赖 B 的结果，但 B 还没返回
- [ ] 超时处理：等待响应超时后的状态清理

**状态边界：**
- [ ] 跨场景：玩家切换场景时，进行中的操作如何处理？
- [ ] 玩家下线：操作进行中玩家下线，数据是否安全保存？
- [ ] 服务器重启：内存中的临时状态是否有持久化？
- [ ] 多端同步：同一账号多设备登录时的冲突处理

---

#### 维度 3：协议设计审查

- [ ] 请求/响应消息的字段是否完整？有没有遗漏必要字段？
- [ ] 错误码是否覆盖了所有失败场景？每个错误码是否有明确的触发条件？
- [ ] 推送消息（Ntf）是否覆盖了所有需要实时通知的场景？
- [ ] 消息大小是否合理？是否有可能产生过大的消息（如返回完整列表而非增量）？
- [ ] 分页/增量同步：大数据量场景是否考虑了分页？
- [ ] 幂等性：同一请求重复发送是否安全？

---

#### 维度 4：服务端设计审查

- [ ] Actor 职责是否清晰？有没有一个 Actor 承担过多职责？
- [ ] Actor 之间的通信是否都通过消息传递？（宪法第五条）
- [ ] 数据持久化方案是否合理？MongoDB/Redis 选型是否正确？
- [ ] 是否有潜在的数据竞争或死锁？
- [ ] 定时器/周期任务是否有正确的生命周期管理？
- [ ] 配置热重载：配置变更后是否需要重启？能否热更新？
- [ ] 内存泄漏：临时数据结构（map、channel）是否有清理机制？

---

#### 维度 5：客户端设计审查

- [ ] Manager 是否遵循 ManagerCenter 架构？（宪法第二条）
- [ ] 事件订阅是否配对（Subscribe/Unsubscribe）？（宪法第三条）
- [ ] 异步操作是否使用 UniTask 且支持取消？（宪法第四条）
- [ ] 网络请求 Result 是否都检查了 IsOk()？（宪法第五条）
- [ ] 资源加载是否通过 LoaderManager 且有释放？（宪法第八条）
- [ ] UI 状态管理：网络请求期间 UI 如何表现？loading 状态？
- [ ] 弱网体验：延迟高时用户操作是否有即时反馈（乐观更新 or loading）？
- [ ] 客户端数据缓存：哪些数据需要本地缓存？缓存失效策略？

---

#### 维度 6：安全与防滥用

- [ ] 服务器权威：所有关键判定是否在服务端执行？客户端是否只做表现？
- [ ] 数值校验：服务端是否验证了客户端发来的所有数值？
- [ ] 频率限制：是否有防刷机制？同一操作的最小间隔？
- [ ] 资源安全：涉及货币/物品变动的操作是否有事务保护？
- [ ] 信息泄露：返回给客户端的数据是否包含不应暴露的信息？（如其他玩家的隐私数据）

---

#### 维度 7：可测试性与可观测性

- [ ] 关键操作是否有日志？日志信息是否足够定位问题？
- [ ] 测试用例是否覆盖了所有主要流程和边界场景？
- [ ] 是否有 GM 指令支持调试？（如快速触发某个状态）
- [ ] 监控指标：是否需要新增监控指标？（如操作次数、延迟、失败率）
- [ ] 灰度策略：新功能是否支持按比例开放？

### 第三步：生成 Review 报告

按以下格式输出：

```markdown
# Plan Review 报告

## 📊 总评

| 维度 | 评级 | 问题数 |
|------|------|--------|
| 需求覆盖度 | ✅ 完善 / ⚠️ 有遗漏 / ❌ 严重缺失 | N |
| 边界条件 | ... | N |
| 协议设计 | ... | N |
| 服务端设计 | ... | N |
| 客户端设计 | ... | N |
| 安全防滥用 | ... | N |
| 可测试性 | ... | N |

**总体评价**：一句话总结 plan 的质量和主要风险。

## 🔴 必须修复（Critical）

> 不修复会导致功能不可用、数据丢失或安全漏洞

### C1: [问题标题]
- **位置**：plan.md 第X节 / plan/server.md 第Y节
- **问题**：具体描述
- **影响**：会导致什么后果
- **建议**：具体的修复方案（给出足够的细节，可以直接写入 plan）

### C2: ...

## 🟡 建议修复（Important）

> 不修复可能导致边界场景 bug 或体验问题

### I1: [问题标题]
- **位置**：...
- **问题**：...
- **建议**：...

### I2: ...

## 🟢 可选优化（Nice to have）

> 提升代码质量、可维护性或用户体验

### N1: [问题标题]
- **建议**：...

## 📝 遗漏场景清单

列出 plan 中**完全没有提及**的场景，每个场景用一句话描述需要补充的设计：

1. [场景描述] — 需要在 [plan 章节] 中补充 [什么内容]
2. ...

## ✅ 做得好的地方

列出 plan 中设计得特别好的 2-3 个点，说明为什么好（正向反馈有助于保持好的设计习惯）。
```

### 第四步：交互式修复

报告输出后，询问用户：

> Review 完成。有 X 个 Critical、Y 个 Important、Z 个 Nice-to-have 问题。
>
> 你想怎么处理？
> 1. **全部修复**：我直接修改 plan 文件，补充所有 Critical + Important 问题
> 2. **只修 Critical**：只修改必须修复的问题
> 3. **选择性修复**：告诉我你想修哪些（如 C1, I2, I3）
> 4. **仅参考**：不修改 plan，我自己来处理

用户选择后：
- 如果需要修改 plan，直接编辑对应文件，在修改处添加注释说明变更原因
- 修改完成后，对修改部分再做一次快速自检，确保新增内容与原 plan 一致无矛盾
- 输出修改摘要（改了哪些文件、改了什么）
