## 参数解析

从 $ARGUMENTS 中解析用户输入，支持以下形式：

- **Bug 描述文本**：自然语言描述的 bug 现象，如 `"NPC 移动后不播放动画"`
- **日志片段**：包含错误日志的文本，如 `"E0217 12:47:12 [behavior_nodes.go:163] panic: nil pointer"`
- **截图路径**：错误截图的文件路径，如 `screenshots/bug-001.png`

调用示例：
```
/dev-debug NPC 在 meeting 状态下不移动
/dev-debug E0217 12:47:12 [behavior_nodes.go:163] panic: nil pointer dereference
/dev-debug /tmp/error-screenshot.png
```

## 工程关系模型

调试时需要明确 bug 所属的工程，不同工程的日志位置、构建方式、代码规范各不相同。

```
workspace/server/                    ← 工作目录（CWD）
├── P1GoServer/                      ← 业务工程（Go 游戏服务器）
│   ├── servers/                        各微服务代码
│   ├── common/                         共享库
│   ├── common/config/                  游戏配置（从配置工程生成）
│   ├── resources/proto/                协议生成的代码（git submodule）
│   ├── bin/log/                        主日志目录（glog）
│   ├── log/err/                        stderr 输出（ERROR 级别）
│   ├── log/out/                        stdout 输出（全量镜像）
│   └── .claude/rules/                  项目级 Rules
│
├── server_old/                      ← Rust 遗留工程（仅供参考，不修改）
│
├── proto/old_proto/                 ← 协议工程（Protocol Buffer 定义）
│   ├── scene/                          场景协议
│   ├── logic/                          逻辑协议
│   └── module.proto                    模块定义
│
└── config/RawTables/                ← 配置工程（游戏配置表 Excel/JSON）
    ├── TownTask/                       任务配置
    ├── TownNpc/                        NPC 配置
    └── ...                             其他配置表
```

### 工程与 Bug 类型的对应关系

| 工程 | 路径 | 典型 Bug | 日志/构建 |
|------|------|----------|-----------|
| 业务工程 | `P1GoServer/` | 运行时 panic、逻辑错误、性能问题 | 日志：`P1GoServer/bin/log/`，构建：`make build/test` |
| 协议工程 | `proto/old_proto/` | 消息字段不匹配、序列化错误 | 需重新 protoc 生成 |
| 配置工程 | `config/RawTables/` | 配置值异常、加载失败 | 需重新运行配置生成工具 |
| Rust 遗留 | `server_old/` | 仅用于对比参考历史实现 | 不修改、不构建 |

### 调试路径规则

- **代码修改**：只在 `P1GoServer/` 中修改，除非 bug 根因在协议/配置工程
- **日志查看**：始终在 `P1GoServer/bin/log/` 下查看，参照 `DEBUG.md` 的进程列表
- **Rules 遵守**：修改 `P1GoServer/` 代码时，必须遵守 `P1GoServer/.claude/rules/` 下的规范
- **跨工程 bug**：明确标注 bug 涉及哪些工程，修复时按工程分别处理

## Phase 0：记忆查询（前置步骤）

在正式分析之前，查询历史 bug 经验，避免重复踩坑。

### 0.1 提取搜索关键词

从用户输入的 bug 描述中提取 2-3 个核心关键词：
- 模块名（如 `BTree`、`NPC`、`Meeting`）
- 错误类型（如 `panic`、`nil pointer`、`deadlock`）
- 功能名（如 `MoveBehavior`、`DialogBehavior`）

### 0.2 搜索历史记忆

使用 claude-mem 的 MCP 工具执行三层搜索：

**第 1 层：索引搜索** — 快速发现相关记录

```
search(query="<关键词1> bug", limit=10)
search(query="<关键词2> fix", limit=10)
```

**第 2 层：时间线上下文** — 对命中结果获取前后文

```
timeline(anchor=<id>, depth_before=3, depth_after=3)
```

**第 3 层：获取详情** — 只对确认相关的记录拉取完整内容

```
get_observations(ids=[<id1>, <id2>])
```

### 0.3 查阅项目记忆

除 claude-mem 外，还需查阅：
- **Auto Memory**（`MEMORY.md`）中的"已知问题"章节
- **DEBUG.md**（`../dev-workflow/DEBUG.md`）中的"常见调试场景"

### 0.4 输出记忆摘要

如果有命中相关记录：

```
历史 bug 经验：
- [来源] 曾遇到过类似的 XXX 问题，根因是 YYY
- [来源] 解决方案：ZZZ
- 需要注意的坑：...
```

如果没有命中，输出"未找到相关历史 bug 经验"并继续。

### 容错规则

- 如果 claude-mem MCP 工具不可用，**跳过 claude-mem 搜索**，仅查阅项目记忆
- Phase 0 不阻塞主流程，搜索失败不影响后续阶段

## Phase 1：Bug 分析

### 1.1 解析用户输入

根据输入类型进行解析：

| 输入类型 | 解析方式 |
|----------|----------|
| Bug 描述文本 | 提取关键实体（模块、NPC、场景）和现象描述 |
| 日志片段 | 提取错误级别、源文件、行号、错误信息、entity_id |
| 截图路径 | 使用 Read 工具查看截图，提取可见的错误信息 |

### 1.2 日志分析

参照 `../dev-workflow/DEBUG.md` 的日志系统说明：

1. **定位相关进程**：根据 bug 类型判断应查看哪个进程的日志
   - NPC 行为/BT/ECS → `scene` 进程
   - 游戏逻辑 → `logic` 进程
   - 数据存储 → `db` 进程
   - 其他：参照 DEBUG.md 1.3 进程列表

2. **查看最新日志**：
   ```bash
   # 查看最近的错误日志
   tail -100 P1GoServer/bin/log/scene.ERROR.log
   # 查看最近的全量日志
   tail -500 P1GoServer/bin/log/scene.INFO.log
   ```

3. **过滤关键信息**：
   - 按 entity_id 过滤：`grep "entity_id=XXX" scene.INFO.log`
   - 按模块过滤：`grep "\[ModuleName\]" scene.INFO.log`
   - 按时间段过滤：`grep "^E0217 12:4" scene.INFO.log`

### 1.3 错误分类

将 bug 归类到以下类型之一：

| 类型 | 特征 | 典型处理方式 |
|------|------|-------------|
| 编译错误 | `make build` 失败 | 修复语法/类型错误 |
| 运行时 panic | `panic:` / `runtime error:` | 添加 nil 检查、边界检查 |
| 逻辑错误 | 行为不符合预期，无报错 | 追踪数据流和状态变更 |
| 性能问题 | 高延迟、高 CPU/内存 | profiling 分析热点 |
| 数据异常 | 数据不一致、丢失 | 检查存储和同步逻辑 |
| 配置错误 | 配置加载失败或值异常 | 检查配置文件和加载逻辑 |

### 1.4 影响范围评估

评估 bug 的影响范围：
- **涉及模块**：哪些包/文件可能相关
- **涉及进程**：哪些服务受影响
- **涉及工程**：业务工程/协议工程/配置工程
- **严重程度**：是否影响核心流程、是否有数据风险

### 1.5 已知问题匹配

结合 Phase 0 的历史经验，判断是否为已知问题：
- 如果是已知问题且有解决方案 → 直接引用方案，跳到 Phase 3
- 如果是已知问题的变体 → 标注关联，继续 Phase 2
- 如果是全新问题 → 继续 Phase 2

### 1.6 输出分析报告

```
Bug 分析报告：
- 错误类型：[编译错误/panic/逻辑错误/性能问题/数据异常/配置错误]
- 现象描述：[用户描述 + 日志证据]
- 相关模块：[module1, module2]
- 相关进程：[scene/logic/db/...]
- 严重程度：[高/中/低]
- 已知问题匹配：[是/否，如是附上历史解决方案]
- 初步假设：[根因猜测 1-3 个]
```

**等待用户确认后进入 Phase 2。**

## Phase 2：Bug 定位

### 2.1 代码搜索

根据分析报告，使用 Explore subagent 搜索相关代码：

1. **错误点定位**：如果有堆栈信息，直接定位到出错文件和行号
2. **关键词搜索**：搜索错误信息中的关键函数名、变量名
3. **模块结构理解**：理解相关模块的整体架构和调用关系

### 2.2 调用链追踪

从错误点出发，向上和向下追踪调用链：

- **向上追踪**（谁调用了出错函数）：使用 LSP 的 `incomingCalls` 或 `findReferences`
- **向下追踪**（出错函数调用了谁）：使用 LSP 的 `outgoingCalls` 或直接阅读代码
- **数据流追踪**：关键变量从哪里来、到哪里去

### 2.3 日志辅助定位

结合 `../dev-workflow/DEBUG.md` 的日志过滤方法：

- 按 `entity_id` 追踪单个 NPC 的完整行为链
- 按模块前缀（如 `[BTree]`、`[Decision]`）过滤特定系统的日志
- 按时间范围缩小搜索范围，找出 bug 触发前后的关键事件

### 2.4 复现条件分析

梳理触发 bug 的条件和步骤：

1. **前置条件**：什么状态下会触发（特定 NPC 类型、特定 Plan、特定时间段等）
2. **触发动作**：什么操作触发了 bug（状态切换、外部事件、定时器等）
3. **必现/偶现**：是否每次都能复现，还是概率性发生

### 2.5 确定根因

区分直接原因和根本原因：

- **直接原因**：代码层面哪一行出了问题（如 nil pointer）
- **根本原因**：为什么会出现这个问题（如初始化遗漏、时序错误、竞态条件）
- **修复方向**：应该修直接原因还是根本原因

### 2.6 输出定位报告

```
Bug 定位报告：
- 根因：[root cause description]
- 直接原因：[direct cause，具体文件:行号]
- 涉及文件：
  * file1.go:L123 — [描述]
  * file2.go:L456 — [描述]
- 复现条件：[触发步骤]
- 修复方向：[方案概述]
- 影响评估：修复是否可能引入新问题
```

**等待用户确认后进入 Phase 3。**

## Phase 3：Bug 修复

### 3.1 修复方案制定

遵循最小改动原则：

1. **优先修根因**：不做临时绕过（workaround），除非紧急且根因修复周期长
2. **最小变更集**：只修改必要的代码，不顺手重构
3. **防御性编程**：在修复点添加必要的防御检查（nil guard、边界检查等）
4. **向后兼容**：确保修复不破坏现有行为

### 3.2 实现修复

按修复方案逐个文件修改：

- 使用 Edit 工具修改代码
- 每次修改后简要说明改动意图
- 如果涉及多个文件，标注修改顺序和依赖关系

### 3.3 补充测试

参照 `../dev-workflow/TEST.md` 的测试规范：

1. **复现测试**：编写能复现原始 bug 的测试用例（修复前应失败）
2. **修复验证测试**：确认修复后测试通过
3. **边界测试**：覆盖相关边界条件
4. **回归保护**：确保已有测试不受影响

### 3.4 构建验证

```bash
# 构建
cd P1GoServer && make build

# 运行测试
cd P1GoServer && make test

# 如果涉及特定包的测试
cd P1GoServer && go test ./path/to/package/... -v -run TestXxx
```

### 3.5 回归验证

确保修复不引入新问题：

- [ ] `make build` 构建通过
- [ ] `make test` 全部测试通过
- [ ] 相关模块的集成测试通过
- [ ] 新增的复现测试通过

### 3.6 输出修复摘要

```
Bug 修复摘要：
- 根因：[root cause]
- 修复方案：[approach]
- 修改文件：
  * file1.go — [改动描述]
  * file2.go — [改动描述]
- 新增测试：
  * test_file_test.go — TestXxx [测试描述]
- 构建验证：通过/失败
- 测试验证：通过/失败
```

**等待用户确认后进入 Phase 4。**

## Phase 4：Bug 经验沉淀

### 4.1 评估沉淀价值

判断本次 bug 是否值得沉淀：

| 条件 | 是否沉淀 |
|------|----------|
| 花费超过 30 分钟定位 | 是 |
| 涉及隐含的系统契约或时序依赖 | 是 |
| 可能在其他地方重复出现 | 是 |
| 简单的拼写/语法错误 | 否 |
| 一次性配置问题 | 否 |

如果不值得沉淀，跳过本阶段，输出"本次 bug 为简单问题，无需沉淀"并结束。

### 4.2 分类沉淀

根据经验类型，沉淀到对应的文档（复用 dev-workflow 的沉淀文档体系）：

| 经验类型 | 目标文档 | 示例 |
|----------|----------|------|
| 调试技巧 | `../dev-workflow/DEBUG.md` | 新的日志过滤方法、定位技巧 |
| 编码规范 | `P1GoServer/.claude/rules/` | 必须 nil 检查的场景、初始化契约 |
| 测试覆盖 | `../dev-workflow/TEST.md` | 新的测试场景、边界条件 |
| 审查清单 | `../dev-workflow/REVIEW.md` | 新的检查项 |
| 系统已知问题 | Auto Memory (`MEMORY.md`) | 待修复的已知缺陷 |

### 4.3 执行沉淀

1. **更新目标文档**：使用 Edit 工具追加经验内容
2. **更新 claude-mem**（如可用）：持久化到跨会话记忆
3. **更新 Auto Memory**（如为系统级已知问题）：更新 `MEMORY.md`

### 4.4 Skill 自优化

回顾本次调试全流程，评估 SKILL.md 自身是否需要优化。检查以下维度：

| 维度 | 检查内容 | 优化动作 |
|------|----------|----------|
| 流程缺失 | 本次调试中是否有某个步骤不在当前流程中，但实际很关键？ | 在对应 Phase 中补充步骤 |
| 流程冗余 | 是否有某个步骤对本类 bug 完全无用，浪费了时间？ | 标注为可选或调整触发条件 |
| 分类不足 | 错误分类表（1.3）是否缺少本次 bug 的类型？ | 新增分类行 |
| 工程映射缺失 | 工程与 Bug 类型对应表是否缺少本次场景？ | 补充对应关系 |
| 辅助文档缺失 | 调试过程中是否需要参考某个文档但辅助文档表中没有？ | 新增辅助文档引用 |
| 定位技巧 | 本次用到的定位方法（如特定日志过滤、LSP 操作）是否值得模板化？ | 补充到 Phase 2 |
| 历史案例 | 本次 bug 是否具有代表性，可作为案例追加到本文件末尾？ | 追加到"调试案例"章节 |

**执行规则**：
- 只在确认有改进价值时才修改 `SKILL.md`，不为优化而优化
- 每次修改附带简短注释说明改进原因（如 `<!-- 2026-02-28: 新增 XX 类型，来源于 YY bug -->`)
- 修改后读取 SKILL.md 确认格式完整，不破坏现有结构

### 4.5 询问用户

询问用户：
- 是否需要新增 Rules？
- 是否需要更新其他辅助文档？
- 是否有其他需要记录的经验？
- 对本次调试流程有无改进建议？（用于 Skill 自优化）

## 执行原则

- **每个 Phase 完成后暂停**，输出摘要等待用户确认再继续
- **最小改动原则**：只修复 bug，不顺手重构
- **证据驱动**：每个结论都要有日志/代码证据支撑
- **已知问题优先匹配**：先查历史经验，避免重复踩坑
- **遇到阻塞立即汇报**：不自行绕过或猜测
