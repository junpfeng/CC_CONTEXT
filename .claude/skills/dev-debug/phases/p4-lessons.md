## Phase 4：Bug 经验沉淀

### 4.1 评估沉淀价值

判断本次 bug 是否值得沉淀：

| 条件 | 是否沉淀 |
|------|----------|
| 花费超过 30 分钟定位 | 是 |
| 涉及隐含的系统契约或时序依赖 | 是 |
| 可能在其他地方重复出现 | 是 |
| 简单的拼写/语法错误 | 否 |
| 一次性配置问题 | 否 |

如果不值得沉淀，跳过本阶段，输出"本次 bug 为简单问题，无需沉淀"并结束。

### 4.2 分类沉淀

根据经验类型，沉淀到对应位置：

| 经验类型 | 沉淀目标 | 示例 |
|----------|----------|------|
| 调试技巧 | 项目调试文档 | 新的日志过滤方法、定位技巧 |
| 编码规范 | 对应子工程的 `.claude/rules/` | 必须 nil 检查的场景、初始化契约 |
| 测试覆盖 | 项目测试文档 | 新的测试场景、边界条件 |
| 审查清单 | 项目审查文档 | 新的检查项 |
| 系统已知问题 | Auto Memory (`MEMORY.md`) | 待修复的已知缺陷 |
| 错误模式 | `docs/knowledge/error-patterns.md` | 新发现的典型错误模式（症状→根因→解法） |
| **MCP 测试案例** | `p3-fix.md` 案例库 | 本次实测中有效的 MCP 验证方法 |

### 4.3 执行沉淀

1. **更新目标文档**：使用 Edit 工具追加经验内容
2. **更新 Auto Memory**（如为系统级已知问题）：更新 `MEMORY.md`

#### 4.3.2 经验沉淀索引追加

将本次沉淀的经验追加到 `docs/knowledge/consolidation-index.md` 对应 section：
- 新增 error pattern → 追加到 `## Error Patterns` section：`- [EP-NNN] {标题} → docs/knowledge/error-patterns.md`
- 新增 .claude/rules/ 规则 → 追加到 `## Coding Rules` section：`- [rule] {标题} → {规则文件路径}`
- 新增 review 检查项 → 追加到 `## Review Checklist Additions` section：`- [RC-NNN] {检查项}（来源：dev-debug P4）`

> 若 consolidation-index.md 不存在则跳过（不阻塞主流程）。

#### 4.3.1 MCP 测试案例回流

若 Phase 3.6 游戏内实测中产生了**可复用的** MCP 测试方法，将其沉淀到 `p3-fix.md` 的「MCP 模拟测试案例库」章节。

**回流条件**（满足任一即可）：

| 条件 | 说明 |
|------|------|
| 新系统/模块 | 案例库中尚无该系统的测试案例（如交通系统、动物系统、载具系统） |
| 新测试手法 | 使用了案例库中未覆盖的验证技巧（如多帧采样、跨系统联动验证、协议收发模拟） |
| 踩坑修正 | 现有案例的写法有坑（如类型引用方式、API 变更），本次实测发现了更可靠的写法 |

**回流格式**（与现有案例保持一致）：

```markdown
##### 案例 N：{系统名}测试（{测试要点}）
<!-- {日期}: 来源于 {bug 简述} -->

\```csharp
// {脚本用途说明}
using UnityEngine;
public class Script
{
    public static object Main()
    {
        // 实际验证代码
    }
}
\```

> **验证要点**：{关键步骤 → 预期结果}
```

**执行规则**：
- 从实测中提取**通用化**后的代码，去除 bug 特定的硬编码值，替换为注释占位符
- 一个案例聚焦一个系统/模块，不混合多个系统的测试逻辑
- 若是对现有案例的修正/增强，直接 Edit 更新原案例而非新增

### 4.4 Skill 自优化

回顾本次调试全流程，评估 SKILL.md 自身是否需要优化。检查以下维度：

| 维度 | 检查内容 | 优化动作 |
|------|----------|----------|
| 流程缺失 | 本次调试中是否有某个步骤不在当前流程中，但实际很关键？ | 在对应 Phase 中补充步骤 |
| 流程冗余 | 是否有某个步骤对本类 bug 完全无用，浪费了时间？ | 标注为可选或调整触发条件 |
| 分类不足 | 错误分类表（1.2）是否缺少本次 bug 的类型？ | 新增分类行 |
| 工程映射缺失 | 工程与 Bug 类型对应表是否缺少本次场景？ | 补充对应关系 |
| 辅助文档缺失 | 调试过程中是否需要参考某个文档但领域知识中没有？ | 新增领域文档引用 |
| 定位技巧 | 本次用到的定位方法（如特定日志过滤、LSP 操作）是否值得模板化？ | 补充到 Phase 2 |
| 历史案例 | 本次 bug 是否具有代表性，可作为案例追加到本文件末尾？ | 追加到"调试案例"章节 |
| 采集源反馈 | 本次采集中哪些源提供了关键线索？哪些源无用？ | 调整 Phase 1 的源优先级或触发条件 |
| **MCP 测试案例** | Phase 3.6 实测中是否产生了新的、可复用的 MCP 测试方法？ | 回流到 `p3-fix.md` 案例库（见 4.3.1） |

**执行规则**：
- 只在确认有改进价值时才修改 `SKILL.md`，不为优化而优化
- 每次修改附带简短注释说明改进原因（如 `<!-- 2026-02-28: 新增 XX 类型，来源于 YY bug -->`)
- 修改后读取 SKILL.md 确认格式完整，不破坏现有结构

### 4.5 经验固化为 Rule/Lesson

如果本次 bug 的根因属于**可机械检查的模式**（编码规范违规、生命周期遗漏、契约未遵守等），评估是否固化为 `.claude/rules/` 下的 lesson 文件：

**固化条件**（满足任一）：
- 同类错误在 error-patterns.md 中出现 ≥2 次
- 根因涉及隐含契约（如"删除实体必须同时调 RemoveEntity"），且无现有 rule 覆盖
- 修复过程中发现现有 rule 表述不完整，需补充

**固化格式**（参考 `auto-work-lesson-*.md`）：
```markdown
# {规则标题}

## 触发条件
当编写或修改 {场景描述} 时触发

## 规则内容
1. {具体规则}
2. {扫描/验证命令}

## 来源
{bug 描述}，根因：{一句话根因}
```

**执行**：写入 `P1GoServer/.claude/rules/` 或 `freelifeclient/.claude/rules/`（按 bug 所在端决定），文件名 `auto-work-lesson-{NNN}.md`。

### 4.6 Bug 文档状态更新

**条件**：仅当 `--caller` 为 `direct`（或未指定）时执行。
`--caller bug-explore` 或 `--caller new-feature` 时跳过本节——归档由调用方负责。

如果本次 bug 对应 `docs/bugs/` 下的已归档条目，自动更新追踪状态：

1. 在 `docs/bugs/{version}/{feature}/fixed.md` 追加修复记录：
   ```
   ## Bug #{N} — {一句话描述}
   - 根因：{root cause}
   - 修复：{改了哪些文件}
   - 验证：{编译/测试/MCP实测 结果}
   - 日期：{当前日期}
   ```
2. 在 `docs/bugs/{version}/{feature}/{feature}.md` 将对应 `- [ ]` 改为 `- [x]`

**如果 bug 不在 docs/bugs/ 中**（如直接 `/dev-debug` 传入描述），跳过此步骤。

### 4.7 自动完成

根据 4.1-4.6 的判断自动执行沉淀，无需询问用户。输出最终摘要：修复内容、沉淀结果、Skill 自优化（如有）、文档更新（如有）。
