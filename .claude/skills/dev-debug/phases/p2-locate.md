## Phase 2：Bug 定位

### 2.0-A Acceptance 模式快速定位

若 `--mode acceptance`，跳过 2.0-2.4 的标准定位流程，改为基于 P1 的 spec-vs-code 对比报告：

1. **`missing_implementation`**：定位规格要求的功能应插入的文件和位置（基于同模块已有实现的模式）
2. **`wrong_implementation`**：列出规格描述与实际代码的具体差异行（字段名/类型/逻辑分支/算法）
3. **`incomplete_implementation`**：列出缺失的边界处理、错误处理、配置项，标注应补充的位置

输出格式与标准 P2 定位报告一致（根因 + 受影响文件列表 + 修复方向），直接进入 P3 修复循环。

---

### 2.0 利用采集数据

在开始代码搜索前，先回顾 Phase 1 的采集结果：

- **日志线索**：从 srv_log / unity_console 中提取的错误堆栈、关键 ID、时间戳
- **变更线索**：从 git_recent 中识别的近期相关改动
- **运行时线索**：从 editor_state / component_data 等获取的异常状态
- **模式匹配**：Phase 0 命中的错误模式（EP 条目）提供的排查方向

> 优先从采集数据中已有的线索出发，避免重复读取已采集的日志和状态。

### 2.1 代码搜索

根据分析报告，使用 Explore subagent 搜索相关代码：

1. **错误点定位**：如果有堆栈信息，直接定位到出错文件和行号
2. **关键词搜索**：搜索错误信息中的关键函数名、变量名
3. **模块结构理解**：理解相关模块的整体架构和调用关系

### 2.2 调用链追踪

从错误点出发，向上和向下追踪调用链：

- **向上追踪**（谁调用了出错函数）：使用 LSP 的 `incomingCalls` 或 `findReferences`
- **向下追踪**（出错函数调用了谁）：使用 LSP 的 `outgoingCalls` 或直接阅读代码
- **数据流追踪**：关键变量从哪里来、到哪里去

### 2.3 日志辅助定位

如果 Phase 1 已采集相关日志，直接使用采集结果，无需重复读取。仅在需要更细粒度过滤时才重新读取日志文件。

结合项目调试文档（如有）的日志过滤方法：

- 按关键 ID 追踪单个实体的完整行为链
- 按模块前缀过滤特定系统的日志
- 按时间范围缩小搜索范围，找出 bug 触发前后的关键事件

### 2.4 复现条件分析

梳理触发 bug 的条件和步骤：

1. **前置条件**：什么状态下会触发（特定配置、特定输入、特定时间段等）
2. **触发动作**：什么操作触发了 bug（状态切换、外部事件、定时器等）
3. **必现/偶现**：是否每次都能复现，还是概率性发生

### 2.5 确定根因

区分直接原因和根本原因：

- **直接原因**：代码层面哪一行出了问题（如 nil pointer）
- **根本原因**：为什么会出现这个问题（如初始化遗漏、时序错误、竞态条件）
- **修复方向**：应该修直接原因还是根本原因

### 2.6 输出定位报告

```
Bug 定位报告：
- 根因：[root cause description]
- 直接原因：[direct cause，具体文件:行号]
- 涉及文件：
  * file1:L123 — [描述]
  * file2:L456 — [描述]
- 复现条件：[触发步骤]
- 修复方向：[方案概述]
- 影响评估：修复是否可能引入新问题
```

**自动进入 Phase 3。**
