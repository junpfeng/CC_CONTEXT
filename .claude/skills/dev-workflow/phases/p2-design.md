# Phase 2：技术设计

> 领域依赖：无

## 2.0 读取结构化需求

Read P1 输出的 `requirements.json`，逐条检查 requirements 和 acceptance_criteria。设计必须覆盖每个 REQ-XXX，遗漏在自审中会被标记。

若 classification=research，还需读取 research_conclusion 作为技术决策依据。

### 锁定决策约束（来自 new-feature idea.md）

若需求文档（或其上游 idea.md）包含 `### 锁定决策` 子节，则逐条读取并作为**硬约束**：
- 锁定决策中的协议定义、数据结构、核心算法、架构选型**不得重新设计**，直接纳入 2.1-2.4 对应章节
- 仅对 `### 待细化` 子节中的项目做补充设计（调用链、边界处理、错误恢复等）
- 若无 `### 锁定决策` 子节（旧格式 idea.md 或直接传入的需求文档），按现有逻辑全量设计

## 2.1 架构设计

1. **系统边界**：明确各功能在哪个工程实现
2. **接口设计**：API / RPC 接口、数据库 Schema、配置结构
3. **状态流转**：核心业务状态机（如有）
4. **错误处理**：错误码和异常场景

## 2.2 详细设计

分工程设计，根据工程类型关注以下要点：

**业务工程**：模块划分、数据结构（定义 + 序列化 + 持久化）、接口定义（API / RPC / 事件回调）、并发控制（锁粒度 + 死锁评估）

**接口/协议工程**（如有独立的 IDL / Schema 定义工程）：新增消息或接口定义、枚举常量、版本兼容性

**配置工程**（如有）：配置项设计（字段 / 类型 / 约束 / 关系）、配置加载和热更新策略

**数据库**：表结构（字段 / 索引 / 外键）、Migration 脚本（DDL + 数据迁移 + 回滚）

**遗留工程参考**：找对标实现、提取设计思路、评估兼容性

> 以上分类并非必须全部涉及，根据项目实际工程结构选取适用项。

## 2.3 事务性设计

涉及多状态变更时必须设计：
- **事务范围**：哪些操作同事务，跨服务补偿机制
- **回滚机制**：失败回滚逻辑和顺序，回滚失败处理
- **幂等性**：幂等键生成规则和检查位置
- **并发控制**：加锁关键路径，锁范围和持有时间，死锁预防

## 2.4 接口契约

定义跨工程契约（根据项目实际工程组成选取适用项）：
- 接口/协议 ↔ 业务：消息格式、版本兼容、错误码映射
- 配置 ↔ 业务：配置项名称类型、更新触发时机、缺失默认行为
- 业务 ↔ DB：事务隔离级别、读写分离策略

## 2.5 客户端验收测试方案

当功能涉及客户端（freelifeclient）时，必须设计 **Unity MCP 验收测试方案**——模拟真人玩家在游戏内的实际操作流程。

### 设计原则

- **真人视角**：每个测试用例 = 一个玩家会做的操作序列（登录 → 打开界面 → 点击按钮 → 验证结果）
- **可观测**：每步操作后明确预期的可观测结果（截图对比、GameObject 状态、日志输出）
- **端到端**：覆盖完整业务流程，不是孤立的单元验证

### 测试用例格式

每个用例包含：

```
[TC-XXX] 用例名称
前置条件：（如：已登录、在某场景、背包有某道具）
操作步骤：
  1. [MCP 操作] 具体操作描述
  2. [MCP 操作] ...
  3. [验证] 预期结果 + 验证方式（screenshot-game-view / gameobject-find / script-execute / console-get-logs）
异常场景：（如：网络断开、重复点击、资源不足时的表现）
```

### MCP 验证手段

| 验证方式 | 适用场景 |
|----------|----------|
| `screenshot-game-view` | UI 显示、场景表现、动画效果 |
| `screenshot-scene-view` | 3D 空间位置、碰撞体、Gizmo |
| `gameobject-find` + `gameobject-component-get` | 对象状态、组件属性值 |
| `script-execute` | 运行时数据读取、复杂条件判断 |
| `console-get-logs` | 错误日志、关键事件日志 |

### 输出

测试方案写入设计文档的「验收测试」章节，供 Phase 5 执行。

## 2.6 输出

生成设计文档（含需求回顾、架构图、各工程设计、事务设计、接口契约、风险缓解、验收测试方案），保存到设计文档目录，记录文档路径供后续审查使用。

## 2.7 自审循环

设计文档生成后，启动自审循环，直到无问题为止：

1. **派发审查 subagent**（`dev-workflow-design-reviewer`），将设计文档路径和需求文档路径传入。审查内容包括但不限于：
   - 架构合理性：模块划分是否清晰、职责是否单一、依赖方向是否正确
   - 协议设计：字段命名、类型选择、版本兼容性、是否有冗余或遗漏
   - DB 设计：索引覆盖、查询模式匹配、数据一致性
   - 事务性：事务边界、回滚完整性、幂等性、并发安全
   - 接口契约：跨工程契约是否完整、错误码是否覆盖所有异常路径
   - 验收测试方案（如有）：用例是否覆盖核心业务流程和异常场景、MCP 操作是否可行、验证方式是否充分
   - 与需求的一致性：是否遗漏需求点、是否引入需求外的复杂度

   reviewer 必须返回：**问题列表**（每项含：`文件路径:行号` 或 `章节名`、严重度 CRITICAL/HIGH/MEDIUM、问题描述、修改建议）和**结论**（`PASS` / `FAIL`）。
2. **主 agent 处理审查结果**：
   - `FAIL` → 主 agent 根据问题列表逐项修改设计文档 → 回到步骤 1 重新审查
   - `PASS` → 审查结束，进入下一步
3. **最多循环 10 轮**。第 10 轮仍 FAIL 时：将剩余问题标注在文档末尾「遗留问题」章节，**仍继续进入 Phase 3**（遗留问题在 P6 会再次审查兜底）

**自审通过（或标注遗留后）自动进入 Phase 3。**
