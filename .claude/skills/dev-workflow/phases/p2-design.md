# Phase 2：技术设计

> 领域依赖：无

## 2.1 架构设计

1. **系统边界**：明确各功能在哪个工程实现
2. **接口设计**：API / RPC 接口、数据库 Schema、配置结构
3. **状态流转**：核心业务状态机（如有）
4. **错误处理**：错误码和异常场景

## 2.2 详细设计

分工程设计，根据工程类型关注以下要点：

**业务工程**：模块划分、数据结构（定义 + 序列化 + 持久化）、接口定义（API / RPC / 事件回调）、并发控制（锁粒度 + 死锁评估）

**接口/协议工程**（如有独立的 IDL / Schema 定义工程）：新增消息或接口定义、枚举常量、版本兼容性

**配置工程**（如有）：配置项设计（字段 / 类型 / 约束 / 关系）、配置加载和热更新策略

**数据库**：表结构（字段 / 索引 / 外键）、Migration 脚本（DDL + 数据迁移 + 回滚）

**遗留工程参考**：找对标实现、提取设计思路、评估兼容性

> 以上分类并非必须全部涉及，根据项目实际工程结构选取适用项。

## 2.3 事务性设计

涉及多状态变更时必须设计：
- **事务范围**：哪些操作同事务，跨服务补偿机制
- **回滚机制**：失败回滚逻辑和顺序，回滚失败处理
- **幂等性**：幂等键生成规则和检查位置
- **并发控制**：加锁关键路径，锁范围和持有时间，死锁预防

## 2.4 接口契约

定义跨工程契约（根据项目实际工程组成选取适用项）：
- 接口/协议 ↔ 业务：消息格式、版本兼容、错误码映射
- 配置 ↔ 业务：配置项名称类型、更新触发时机、缺失默认行为
- 业务 ↔ DB：事务隔离级别、读写分离策略

## 2.5 输出

生成设计文档（含需求回顾、架构图、各工程设计、事务设计、接口契约、风险缓解），保存到设计文档目录。
