# Phase 2：合并整理

> 目标：根据 Phase 1 审查报告，执行文档的合并、更新、删除和重分类。

## 前置条件

- Phase 1 审查报告已生成（`docs/_review_report.md`）
- 用户已确认审查报告中的变更计划

## 变更前快照

执行任何文件变更前，先检查 docs/ 目录是否在 git 仓库中：
- **在 git 仓库中**：提示用户先 commit 当前状态（或确认已 commit），确保可通过 `git checkout` 回滚
- **不在 git 仓库中**：提醒用户当前无版本控制，变更不可自动回滚，需用户明确确认后继续

## 执行策略

按优先级顺序执行，每类操作完成后输出变更摘要：

### 1. 删除完全过时的文档

- 仅删除审查报告中标记为「完全过时」且用户已确认的文档
- **删除前先处理引用方**：根据审查报告的「引用影响汇总」，逐一更新或移除引用方中的链接
- 删除后记录文件名和已更新的引用方到变更日志

### 2. 更新部分过时的文档

- 移除过时章节，保留有效内容
- 更新与代码不一致的描述（使用 Grep/Glob 获取最新代码状态）
- 标记 TODO 已完成的项

### 3. 合并冗余文档

合并规则：
- **内容重复**：保留更完整的版本，补充另一版本的独有内容，删除冗余版本
- **包含关系**：保留大文档，删除被包含的小文档
- **碎片化**：创建合并文档，按逻辑章节组织，删除原碎片

合并时：
- 使用 subagent 执行单个合并任务（每个 subagent 处理一组相关文档）
- 合并后的文档必须保留所有有效信息
- subagent prompt 包含：「去除重复内容，保留所有有效信息，返回合并结果，不超过 1000 字摘要 + 完整合并文档」

### 4. 重分类

- 将放错位置的文档移动到正确子目录
- **安全移动三步法**（防止中间失败导致断链）：
  1. 先执行文件移动（`git mv` 或 Bash `mv`）
  2. 立即更新所有引用方：将旧路径替换为新路径
  3. 使用 Grep 搜索旧路径二次验证无遗漏（Phase 0 引用图可能不完整）
- 对每个引用方文件执行 Edit 替换路径

### 5. 格式统一

- 确保每个文档有清晰的一级标题
- 确保章节结构合理（标题层级不跳跃）
- 移除无意义的占位内容

### 6. 引用同步（贯穿所有操作）

每次删除/合并/移动文档后，立即同步引用方：

| 变更类型 | 引用方处理 |
|---------|-----------|
| 删除 | 移除引用链接，或替换为合并后的目标文档链接 |
| 合并 | 将指向被合并文档的链接更新为合并后文档路径 |
| 移动/重命名 | 将旧路径替换为新路径 |

处理顺序：
- **删除**：先更新引用方（移除或替换链接）→ 再删除文件
- **合并**：先创建合并后文件 → 更新引用方指向新文件 → 删除旧文件
- **移动/重命名**：先移动文件 → 立即更新引用方（短暂断链但可追踪）

## 输出

向用户输出变更摘要：
- 已删除 N 个文件
- 已更新 N 个文件
- 已合并 N 组文件（列出合并详情）
- 已移动 N 个文件
- 已同步 N 处引用（列出每个引用方文件和变更内容）

## 写入规范

- **单次 Write/Edit 不超过 150 行**，超过时拆分为多次工具调用
- 合并大文档时先 Write 骨架（标题+空章节），再逐章节 Edit 填充
- 文件移动：优先用 `git mv`（git 仓库），否则用 Bash `mv`
- 删除文件：用 Bash `rm`，删除前确认文件路径无误

## 约束

- 每个 subagent 的 prompt 中包含返回长度限制（统一 1000 字上限）
- 所有删除/合并操作必须在用户确认审查报告后才执行
- 遇到文件读写失败时记录并跳过，不中断整体流程
