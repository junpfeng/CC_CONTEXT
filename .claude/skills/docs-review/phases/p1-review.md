# Phase 1：审查评估

> 目标：对比代码现状，识别过时、冗余、可合并的文档，产出审查报告。

## 前置条件

- Phase 0 盘点报告已生成（`docs/_review_inventory.md`）

## 审查维度

### 1. 过时检测

对每个文档检查：
- **代码引用是否存在**：文档中提到的文件路径、函数名、类名是否还存在于代码库中
- **架构描述是否匹配**：文档描述的架构与当前代码结构是否一致
- **TODO/待办是否已完成**：文档中的 TODO 项是否已在代码中实现
- **版本/日期标记**：是否有明显过时的日期或版本引用

使用 subagent 分批检查（每个 subagent 检查同一子目录下 2-3 个相关文档）：
- **分批策略**：按子目录分组，每轮最多 3 个并行 subagent，每轮消化结果写入报告后再派发下一轮
- subagent 使用 Grep/Glob 验证代码引用
- 返回格式：「文件名 | 状态(有效/部分过时/完全过时) | 过时内容摘要」
- 每个 subagent prompt 包含：「返回精简结果，不超过 1000 字」

### 2. 冗余检测

识别以下冗余模式：
- **内容重复**：两个文档描述同一系统/功能的相同方面
- **包含关系**：一个文档的内容完全被另一个文档覆盖
- **碎片化**：同一主题散落在多个小文档中，应合并为完整文档

### 3. 分类检查

检查文档是否放在正确的子目录：
- 设计方案 → `design/`
- 领域知识 → `knowledge/`
- 经验总结 → `postmortem/`
- 参考资料 → `reference/`
- 工具文档 → `tools/`

### 4. 质量检查

- 是否缺少标题或结构混乱
- 是否有大段无意义的占位内容
- INDEX.md 是否与实际文件一致

### 5. 引用影响评估

基于 Phase 0 的引用图，对每个建议删除/移动/重命名的文档：
- 列出所有引用方文件及具体行号
- 评估变更对引用方的影响（断链/需更新路径/语义失效）
- 对高引用文档（≥3 处引用）标记为「高影响」，变更需额外谨慎

## 输出

生成审查报告 `docs/_review_report.md`，结构如下：

```markdown
# 文档审查报告

## 过时文档
| 文件 | 状态 | 过时内容 | 引用方数量 | 建议操作 |
|------|------|---------|-----------|---------|

## 冗余/可合并文档
| 文档组 | 冗余类型 | 引用方数量 | 建议操作 |
|--------|---------|-----------|---------|

## 分类调整（移动/重命名）
| 文件 | 当前位置 | 建议位置 | 引用方 | 需同步更新 |
|------|---------|---------|--------|-----------|

## 引用影响汇总
| 变更文档 | 变更类型 | 引用方文件 | 引用行号 | 需更新内容 |
|---------|---------|-----------|---------|-----------|

## 质量问题
| 文件 | 问题 | 建议操作 |
|------|------|---------|
```

在报告开头添加统计摘要：
- 总文档数 / 有效 / 部分过时 / 完全过时
- 冗余组数
- 需重分类数

向用户输出报告摘要，等待确认后进入 Phase 2。
