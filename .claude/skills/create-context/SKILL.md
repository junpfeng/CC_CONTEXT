> 本技能不接受参数。如果用户的意图不明确（处理范围、是否只检查等），直接用 AskUserQuestion 询问。

- **最小上下文原则**：常驻加载的文档只保留最精简内容，其余按需引用；确保文档精简、准确、可维护

## 文档体系

Claude Code 上下文文档分为两层：

- **工作空间级**：`CLAUDE.md`、`.claude/rules/`、`.claude/INDEX.md`、`.claude/skills/`、`docs/`
- **工程级**：各子工程内的 `CLAUDE.md` 和 `.claude/rules/`

> 具体包含哪些子工程，由 Phase 1 自动分析并与用户确认，不在此硬编码。

### 各文档职责

| 文档 | 层级 | 职责 | 关键内容 |
|------|------|------|----------|
| `CLAUDE.md`（workspace） | 工作空间 | 项目总览与导航 | 目录结构、子工程索引、按需加载规则、工作规范 |
| `.claude/rules/constitution.md` | 工作空间 | 最高优先级规则 | 禁止操作、工作流程、子工程索引 |
| `.claude/INDEX.md` | 工作空间 | 目录索引 | rules/skills/settings 的索引 |
| `docs/` | 工作空间 | 项目文档 | 设计方案、经验总结、知识图谱、参考资料 |
| `CLAUDE.md`（工程级） | 工程 | 工程开发指南 | 构建命令、目录结构、技术栈、常用操作 |
| `.claude/rules/*` | 工程 | 工程级强制规则 | 代码风格、架构约束、命名规范 |

### docs/ 目录结构

| 子目录 | 用途 | 说明 |
|--------|------|------|
| `design/` | 设计方案 | 架构设计、功能设计、任务拆解、需求文档 |
| `postmortem/` | 经验总结 | 踩坑复盘、调试记录、事故回顾 |
| `knowledge/` | 知识图谱 | 系统原理、模块关系、架构概览 |
| `reference/` | 参考资料 | 第三方技术、语言特性、工具用法 |

> 详见 `docs/README.md`。新增文档时遵循其命名规范（如 `design-[feature].md`、`postmortem-[topic].md`）。

## 工作流程

### Phase 1：工程识别与现状分析

1. **扫描工作空间结构**：
   - 识别所有顶层目录（潜在的子工程）
   - 检查每个目录是否有独立的构建系统（Makefile、go.mod、Cargo.toml 等）
   - 确认哪些是真正的子工程，哪些是辅助目录（docs、tools 等）

2. **检查现有文档**：
   - 扫描所有 `CLAUDE.md` 文件
   - 扫描所有 `.claude/rules/` 目录
   - 评估每个文档的完整性

3. **输出工程识别结果和文档现状**，格式：

```
识别到的子工程：
  1. <工程名> — <技术栈> — <一句话描述>
  2. ...

辅助目录（非工程）：
  - <目录名> — <说明>

上下文文档现状：

工作空间级：
  [✅/⚠️] CLAUDE.md — [完整/部分/缺失]
  [✅/⚠️] .claude/rules/constitution.md — [完整/部分/缺失]
  [✅/⚠️] .claude/INDEX.md — [完整/部分/缺失]

工程级：
  <工程名>:
    [✅/⚠️] CLAUDE.md — [完整/部分/缺失]
    [✅/⚠️] .claude/rules/ — [N 个规则文件 / 缺失]
  ...

待补充项：
  1. [具体缺失或不完整的文档]
  2. ...
```

4. **等待用户确认**：
   - 确认工程识别是否正确（是否有遗漏或误判）
   - 确认哪些工程需要创建/补充文档
   - **用户确认后才进入 Phase 2**

**如果是 `--check` 模式，到此结束。**

---

### Phase 2：文档生成/补充

> **前置条件**：Phase 1 的工程识别结果已获得用户确认。

根据用户确认的工程列表和文档现状，按以下优先级处理：

#### 2.1 工作空间级文档

**CLAUDE.md**（如需创建或补充）：

需要包含的内容清单：
- [ ] 项目简介（一句话描述）
- [ ] 目录结构表（子工程列表 + 说明 + CLAUDE.md 路径）
- [ ] 按需加载规则（明确指出仅在需要时加载子工程文档）
- [ ] 工作规范（最小上下文原则、代码注释语言等）
- [ ] .claude 目录说明

**constitution.md**（如需创建或补充）：

需要包含的内容清单：
- [ ] 禁止操作（如禁止手动修改自动生成的文件）
- [ ] 工作流程规则（修改前必须查阅 rules 等）
- [ ] 子工程索引表

**INDEX.md**（如需创建或补充）：

需要包含的内容清单：
- [ ] rules 索引
- [ ] skills 索引
- [ ] settings 说明

#### 2.2 工程级文档

**CLAUDE.md**（为每个子工程）：

通过分析工程代码自动提取以下信息：

| 信息项 | 提取方式 |
|--------|----------|
| 技术栈 | 检查 go.mod / Cargo.toml / package.json 等 |
| 构建命令 | 检查 Makefile / build scripts |
| 目录结构 | 分析顶层目录布局 |
| 入口点 | 查找 main 函数 / main 包 |
| 关键依赖 | 从依赖管理文件中提取 |
| 测试方式 | 检查测试脚本和测试目录 |

模板结构：
```markdown
# <工程名>

<一句话描述>

## 技术栈
- 语言：...
- 框架：...
- 关键依赖：...

## 目录结构
| 目录 | 说明 |
|------|------|
| ... | ... |

## 构建与运行
- 构建：`<命令>`
- 测试：`<命令>`
- 运行：`<命令>`

## 开发规范
- [引用 .claude/rules/ 中的规则]

## 常见操作
- ...
```

**`.claude/rules/`**（如需创建）：

根据工程特征决定需要哪些规则文件：

| 规则文件 | 创建条件 | 内容来源 |
|----------|----------|----------|
| `constitution.md` | 所有工程 | 工程特有的禁止操作和强制规则 |
| 代码风格规则 | 有明确编码规范的工程 | 从现有代码模式中提取 |
| 架构规则 | 有特定架构模式的工程 | 从代码结构中分析 |

---

### Phase 3：关联验证

确保工作空间级和工程级文档的一致性：

1. **索引一致性**：workspace CLAUDE.md 中的子工程索引是否覆盖所有工程
2. **路径正确性**：所有文档中引用的路径是否存在
3. **宪法一致性**：工程级 constitution 不与 workspace 宪法冲突
4. **按需加载规则**：确认 workspace CLAUDE.md 中明确标注了按需加载原则

输出验证报告：

```
关联验证结果：
  ✅ 索引覆盖完整
  ✅ 路径引用正确
  ⚠️ [发现的问题及修复建议]
```

**等待用户确认后执行修复。**

## 执行原则

- **分析优先**：先完整分析现状，再决定创建/补充哪些文档
- **最小侵入**：补充已有文档时保持原有结构，仅添加缺失内容；不重写已有内容
- **自动提取**：尽量从代码中自动提取信息，减少用户手动输入
- **每阶段确认**：每个 Phase 完成后暂停，展示结果等待用户确认
- **格式一致**：新建文档遵循工作空间中已有文档的格式惯例
- **不越权**：不修改宪法中列出的自动生成区域
